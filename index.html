1) Research Dataset Recommendation System 25+ sources · ~2h → <5min

Problem: Researchers spend substantial time searching many portals to locate an appropriate dataset.

Constraints: Reproducible; easy to extend with new sources; no invented metrics.

Approach: Curate dataset catalog across 25+ sources (e.g., CFPB, FRED, Census, WRDS); match question to catalog; provide analysis notebook (CFPB) with OLS regression and t-tests.

Architecture:

[Research question] → [Text parsing/keywords] → [Dataset catalog (25+ sources)]
                         ↓
                   [Matching/ranking]
                         ↓
            [Recommendations + access notes] → [Analysis notebook (CFPB example)]

Results: Reduced manual dataset search time from ~2 hours to <5 minutes; demonstrated end-to-end analysis on CFPB consumer complaints using OLS regression and t-tests.

Links: GitHub repo

Next: Add richer query understanding (synonyms/ontologies) and evaluation metrics for recommendation relevance.

2) Canvas Platform Data Ingestion raw/cur/meta · watermarking · DQ

Problem: Analytics teams need reliable, rerunnable ingestion of semi-structured data with auditing and data quality checks.

Constraints: Incremental loads without full reprocessing; run auditing; schema-drift awareness.

Approach: Medallion architecture (raw → curated → meta); watermark-based incremental ingestion; run metadata + data quality checks + schema-drift detection.

Architecture:

[JSONL source] → [raw tables] → [curated transforms] → [analytics-ready tables]
                    ↓               ↓                   ↓
              [meta.job_run]   [meta.dq_results]   [meta.schema_drift]
                    ↓
             [watermark (rerun-safe incremental)]

Results: Reduced reprocessing by loading only new/changed data based on watermark; improved transparency via run auditing + quality monitoring.

Links: GitHub repo

Next: Add alerting for DQ failures and contract tests to guard against upstream schema changes.

3) Wikipedia API Extraction EN↔ZH · multilingual embeddings

Problem: Cross-language content differences can introduce inconsistency; need scalable mismatch discovery.

Constraints: Candidate pairing + similarity at document and sentence levels; no invented accuracy numbers.

Approach: Extract pages; generate EN↔ZH candidate pairs across identity categories; compute similarity using multilingual embeddings; surface low-similarity sentence pairs for review.

Architecture:

[Wikipedia API] → [Page extraction] → [Candidate pairing EN↔ZH]
                       ↓
              [Sentence segmentation] → [Multilingual embeddings] → [Similarity scoring]
                                                                     ↓
                                                             [Low-similarity mismatches]

Links: GitHub repo

Next: Add human-in-the-loop labeling to validate mismatch quality and calibrate thresholds.

4) Crossref Retraction Metadata Analysis 208 DOIs drift (Apr 2023 vs Jul 2024)

Problem: Incorrect/drifting retraction metadata can propagate misinformation in scholarly systems.

Constraints: Snapshot-based comparison; phrase extraction; reproducible reporting.

Approach: Collect DOI metadata snapshots; extract retraction phrases in titles; compare with recorded status; report drift.

Architecture:

[Crossref metadata] → [Snapshot Apr 2023] ┐
                    → [Snapshot Jul 2024] ├→ [Phrase extraction] → [Consistency checks] → [Drift report (208 DOIs)]
                                          ┘

Results: Identified 208 DOIs marked retracted in April 2023 but not in July 2024; highlighted risk of relying on a single metadata source.

Links: GitHub repo

Next: Add multi-source validation (publisher notices / other registries) and anomaly detection for status shifts.
